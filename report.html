Section 01

Platform & Stack

What was considered

Two parallel builds were underway: Clare built a custom CRM from scratch using Claude Code (approximately 40 hours of development), hosted on Vercel with Supabase as the database. Josephine independently built a separate CRM on Lovable, also backed by Supabase. Both were reviewed in a joint session.

ChatGPT was also consulted and recommended Lovable + Supabase as the primary path, characterising Claude Code as slower and better suited to technical founders. It suggested this was a fresh-start decision.

Agreed
Josephine's Lovable build has a superior UI: better dashboard, pipeline intelligence, safety check-in workflow, probability-weighted forecasting, and richer contact fields. Clare's build has superior infrastructure: WhatsApp integration, multi-tenant architecture, automations, flows, practices system, and lead forms.
Recommendation
Export Josephine's Lovable codebase to a private GitHub repository. Host it on the same Vercel + Supabase stack Clare already uses. Migrate Clare's backend features (WhatsApp bridge, practices system, lead forms, automations) into the Lovable UI. This is not a fresh-start decision — it is a merger of two existing builds.
Clare's pushback on ChatGPT
ChatGPT's framing of 'Lovable vs Claude Code' misses the key context: 40 hours of working infrastructure already exist. The question is not which platform to start on — it is how to merge what has been built.
Risk to manage
Lovable is a managed platform. Complex integrations (WhatsApp webhooks, API keys, automations) should be built on infrastructure you control, not inside Lovable's managed layer. Export the code before adding integrations.
Section 02

WhatsApp Integration

What was attempted

Meta Cloud API was set up directly. Issues encountered: 24-hour token expiry on test credentials, phone number already registered to personal WhatsApp (requiring migration), webhook verification failures, and high complexity for non-technical users.

A QR code bridge using Baileys (open source WhatsApp Web library) was then built and deployed locally. Messages sync both to the phone app and the CRM simultaneously. The bridge imports historical messages and handles real-time bidirectional sync.

Clare's non-negotiable
"WhatsApp data MUST come into the app seamlessly from day one for all users." This overrode the standard lean startup advice to validate before building the integration. WhatsApp is itself what needs to be validated — building without it validates a different product.

Options evaluated

Option User friction Meta visibility Cost Verdict
Meta Cloud API (direct) ✗ Very high ✗ Maximum Free ✗ Rejected
360dialog ~ Low (embedded flow) ~ High (BSP) €50/mo (→€10 as partner) ~ Phase 2
Twilio ✗ High (migration required) ~ High Pay per message ✗ Rejected for now
QR Bridge (Baileys) ✓ Very low (scan once) ✓ None ~€20/mo server ✓ Selected (Phase 1)
Telegram Bot ✓ Very low ✓ None Free ✓ Build in parallel
Recommendation
Deploy the QR bridge as a hosted persistent service (Railway or Fly.io). Each user scans once from their CRM settings — no migration, no Meta account, no monthly fee per number. Meta cannot see the platform. When individual number bans become a meaningful operational problem (estimated 200+ active users), migrate to 360dialog partner programme. Build Telegram bot integration in parallel — free, official, zero risk.
Meta risk for sextech
Any official Meta API path creates direct visibility with Meta, which has a documented history of deplatforming sextech businesses. The QR bridge approach inverts this risk: Meta cannot see the platform at all. If they detect and ban a number, it is one user affected, not the entire platform.
Section 03

Payment Processing

ChatGPT recommended Stripe as the default payment processor. This was immediately challenged.

Clare's pushback
"Stripe is often not possible in sextech." Stripe, PayPal, and Square have all deplatformed sex work adjacent businesses, often without warning and with funds held in transit. FOSTA-SESTA made US card networks broadly restrictive toward the industry.
Recommendation
Design the billing layer to be processor-agnostic from day one. Recommended stack: (1) SEPA direct debit for European users — bypasses card networks entirely, low cost, reliable for subscriptions. (2) CCBill or Segpay as the card processor — expensive (10–15% fees) but will not deplatform. (3) Crypto (USDC/Monero) as an option — signals platform literacy to the community.

Section 06

Security & Privacy

This platform handles names, phone numbers, booking history, messages, location data, sexual preferences, and safety information for a population that faces specific risks from data exposure. Security is not a feature — it is a foundation.

Already in place

  • Supabase Row Level Security — each provider's data isolated at database level
  • WhatsApp access tokens encrypted at rest (AES-256-GCM)
  • noindex/nofollow on all public-facing pages
  • Lead form data deletion on rejection (stated in form copy)
  • EU hosting (Supabase Frankfurt)

Recommended additions

  • MFA for provider accounts
  • Data retention policy
  • Delete-on-request workflow (GDPR Article 17)
  • Encrypted backups
  • Security consultant before public launch
Live location: excluded
Live GPS location sharing was considered for the safety check-in feature and excluded on privacy grounds. Location is stored as a city name or address string only.
